# Plan EQUIM - Fase 1

## Objetivo de esta fase

Definir una arquitectura de información simple y corporativa para la nueva versión local de EQUIM sobre la base visual de `tidy-v2`, sin arrastrar el desorden del sitio actual.

La decisión clave de esta fase es esta:

No se va a migrar `equimsas.com` tal como está. Se va a reconstruir el relato comercial e institucional con menos secciones, mejor jerarquía y un tono más técnico.

## Diagnóstico resumido

El sitio actual sí contiene activos valiosos:

- Trayectoria de más de 50 años.
- Enfoque en agroindustria.
- Tres líneas de trabajo reconocibles.
- Muestra de clientes.
- Historial de proyectos.

Pero hoy está debilitado por estos problemas:

- La navegación principal mezcla información estratégica, comercial y sensible.
- El home no prioriza con claridad qué hace EQUIM y por qué confiar.
- Hay bloques que distraen del negocio industrial.
- La experiencia visual transmite sitio heredado, no empresa técnica actual.
- Existen piezas de contenido que parecen añadidas por acumulación y no por criterio editorial.

## Arquitectura objetivo

La estructura propuesta para la nueva versión es:

- Inicio
- Nosotros
- Soluciones
- Proyectos
- Contacto

Esta arquitectura es suficiente para una primera versión seria. No conviene crear más páginas mientras no exista una necesidad real de negocio o SEO.

### Navegación secundaria

`Reorganización` no debe desaparecer, pero tampoco debe convivir con la navegación comercial principal.

La decisión correcta para el nuevo sitio es:

- mantener `Reorganización` como página secundaria,
- enlazarla desde footer o bloque institucional,
- concentrar allí los documentos y referencias formales,
- evitar que interfiera con el recorrido comercial del usuario.

En otras palabras:
`Reorganización` es una necesidad informativa y jurídica, no una puerta principal de venta.

## Estructura propuesta del home

### 1. Hero principal

Objetivo:
Explicar en una sola pantalla qué hace EQUIM, para quién trabaja y cuál es su capacidad principal.

Debe responder tres preguntas:

- Qué hace la empresa.
- En qué sectores o procesos interviene.
- Cómo contactar o ver proyectos.

### 2. Capacidades principales

Bloque corto y muy claro con tres capacidades:

- Diseño
- Fabricación
- Montaje

Este bloque existe para ordenar la propuesta de valor. Hoy esa idea aparece en el sitio, pero dispersa.

### 3. Líneas de solución

Agrupar la oferta en tres frentes:

- Equipos para procesos agroindustriales de cereales.
- Manejo y transporte de granos en puerto marítimo.
- Servicios de izaje, manipulación y montaje.

Estas líneas ya existen en el sitio actual y deben mantenerse, pero mejor presentadas.

### 4. Proyectos destacados

No conviene listar demasiados proyectos en el home. Mejor seleccionar pocos casos representativos.

Objetivo:

- Dar prueba de experiencia real.
- Mostrar escala y tipo de trabajo.
- Llevar a la página de proyectos.

### 5. Clientes

Bloque de logos o nombres institucionales.

Objetivo:

- Construir confianza rápidamente.
- Evitar sobrecargar con demasiados elementos visuales.

### 6. Nosotros / trayectoria

Resumen corto con:

- historia,
- experiencia,
- cobertura,
- enfoque de trabajo.

No hace falta contar toda la empresa en el home. Solo lo suficiente para dar contexto y pasar a la página interna.

### 7. Equipo o liderazgo

Aquí hay que ser críticos:

No conviene inflar esta parte con demasiados perfiles si no hay material fotográfico y textual sólido.

Para la primera versión bastaría con:

- liderazgo principal,
- dirección de proyectos,
- opcionalmente una nota sobre capacidad técnica y operativa.

### 8. Contacto / cierre comercial

Bloque final claro con:

- ubicación,
- teléfonos,
- correo,
- llamada a la acción.

El objetivo no es "decorar el footer", sino cerrar la lectura con una salida comercial concreta.

## Páginas internas propuestas

### Nosotros

Debe concentrar:

- historia de la empresa,
- experiencia acumulada,
- enfoque de trabajo,
- equipo directivo o responsables visibles.

No debería convertirse en una página genérica de relleno.

### Soluciones

Debe organizar la oferta técnica en bloques claros.

Posible estructura:

- Agroindustria de cereales.
- Soluciones para puertos.
- Izaje y montaje.
- Capacidades transversales.

Esta página ayuda a separar el discurso comercial del detalle técnico.

### Proyectos

Debe actuar como vitrina de experiencia real.

Idealmente incluir:

- nombre del proyecto,
- tipo de solución,
- cliente si aplica,
- ubicación si aplica,
- imagen representativa,
- breve descripción.

No conviene heredar una estructura de blog.

Además, el nuevo sistema de proyectos debe quedar pensado para actualización simple.

La recomendación para esta etapa es:

- no replicar el modelo del CMS viejo,
- no depender de posts sueltos con estructura inconsistente,
- centralizar los datos de proyectos en una sola fuente editable,
- separar `proyectos destacados` del home y `catálogo completo` en la página interna.

### Reorganización

Debe existir como página institucional secundaria.

Contenido esperado:

- breve contexto introductorio,
- listado de documentos oficiales,
- nombres claros de cada archivo,
- fechas,
- enlaces de descarga,
- datos de contacto si aplica para temas relacionados.

No debería intentar parecer una sección comercial ni ocupar espacio prioritario en el home.

### Contacto

Debe ser muy simple:

- datos de contacto,
- ciudad / dirección,
- CTA,
- opcionalmente mapa o referencia geográfica.

No necesita elementos de soporte SaaS, formularios complejos ni integraciones innecesarias en esta primera etapa.

## Matriz de contenido del sitio actual

### Conservar

- Mensaje de experiencia de más de 50 años.
- Identidad de grupo empresarial colombiano.
- Las tres líneas de servicio principales.
- Clientes visibles.
- Proyectos históricos aprovechables.
- Información de contacto base.

### Reescribir

- Texto de "Nosotros".
- Presentación de servicios.
- Bloque de equipo.
- Mensajes del hero.
- Descripciones de proyectos destacados.

La razón es simple:
El contenido actual comunica, pero no jerarquiza bien ni tiene una voz corporativa consistente.

### Fusionar

- "Nosotros" + mensaje de trayectoria.
- "Qué hacemos" + líneas de servicio.
- "Nuestro equipo" + liderazgo visible.
- Proyectos sueltos del CMS dentro de una sola página de portafolio.
- Documentos de reorganización dentro de una sola página institucional ordenada.

### Eliminar del nuevo sitio

- Componente de X / Twitter.
- Bloque "Sabías que...".
- Referencia a aliado financiero `Smart Factoring`.
- `Reorganización` en la navegación principal.
- Elementos duplicados de menú o footer heredados del CMS.
- Cualquier página o bloque que se sienta como blog si no aporta negocio.

## Criterio para usar Tidy

La plantilla `Tidy` se usará como sistema visual y estructura base, no como arquitectura final.

Eso implica:

- conservar layout, rejillas, navegación y recursos visuales útiles,
- eliminar páginas tipo SaaS que no aplican,
- adaptar el tono a empresa industrial,
- simplificar la navegación,
- dejar una versión local limpia antes de pensar en repo y despliegue.

## Criterio para reconstruir Proyectos

El sitio original parece mezclar:

- páginas de solución,
- fichas de equipos,
- posts sueltos de proyectos,
- galerías asociadas al CMS.

Eso explica por qué hoy `Proyectos` no carga o no se percibe de forma consistente.

La salida no es reparar ese modelo viejo, sino normalizarlo.

La propuesta es reconstruir `Proyectos` en dos niveles:

### Nivel 1. Catálogo visible para usuarios

Con filtros simples por:

- Agroindustria
- Puertos
- Izaje y montaje

Y cada tarjeta con:

- nombre,
- línea de negocio,
- ubicación,
- resumen breve,
- imagen principal.

### Nivel 2. Fuente única de datos editable

Para que luego sea fácil actualizar, la recomendación técnica inicial es usar un archivo de datos centralizado, por ejemplo `proyectos-data.js`.

La razón para preferir `JS` y no `JSON` en esta etapa local es práctica:

- si el sitio se abre localmente sin servidor, `fetch()` sobre JSON puede fallar o volverse incómodo,
- un archivo JS cargado por script evita esa fricción,
- permite reutilizar los mismos datos en home y en la página de proyectos.

Campos recomendados por proyecto:

- `title`
- `photo`
- `summary`
- `location`
- `client`
- `category`
- `featured`
- `image`

La idea es dejar una estructura que después pueda migrarse a repo, CMS o generador estático sin reescribir el contenido otra vez.

### Estándar mínimo de mantenimiento para proyectos

Para EQUIM, el catálogo de proyectos debe mantenerse con el menor esfuerzo posible.

Por eso se define un estándar corto y obligatorio:

- una foto,
- una descripción rápida,
- una ubicación,
- un cliente.

Y solo dos campos de control adicionales:

- `category`
- `featured`

Esto evita que el catálogo se vuelva difícil de alimentar.

La regla práctica es esta:

- si un proyecto necesita demasiados datos para publicarse, no se publicará nunca,
- si el formato es breve y uniforme, sí se podrá actualizar con constancia.

## Estructura recomendada para `proyectos-data.js`

```js
window.EQUIM_PROJECTS = [
  {
    title: "Tolvas graneleras",
    photo: "./assets/proyectos/tolvas-graneleras.jpg",
    summary: "Sistema para manejo y almacenamiento de graneles en operación portuaria.",
    location: "Puerto Príncipe, Haití",
    client: "Cliente por confirmar",
    category: "puertos",
    featured: true
  }
];
```

Notas de criterio:

- `title` debe ser corto y claro.
- `summary` debe poder leerse en pocos segundos.
- `location` ayuda a dar contexto y credibilidad.
- `client` puede dejarse como "Cliente por confirmar" o similar si no conviene publicarlo aún.
- `category` permite ordenar el catálogo sin tocar el HTML.
- `featured` sirve para decidir qué sale primero en home o en el listado.

## Decisiones ya recomendadas

- No usar X/Twitter en la nueva versión.
- No mostrar `Reorganización` en navegación principal.
- Sí crear una página secundaria de `Reorganización` con documentos.
- No mezclar contenido ambiental o financiero con la propuesta industrial.
- No construir una web con demasiadas páginas desde el inicio.
- No heredar la estructura tipo CMS/blog del sitio actual.
- Reconstruir `Proyectos` desde una fuente única de datos y no desde el modelo viejo del CMS.

## Entregable aprobado para pasar a fase 2

Si esta fase se aprueba, la siguiente ejecución será:

Limpiar `tidy-v2` para dejar solo la estructura mínima de EQUIM:

- `index.html`
- `about.html` o equivalente para `Nosotros`
- una página de `Soluciones`
- una página de `Proyectos`
- `contacto.html`
- una página secundaria de `Reorganización`

Todo lo demás quedaría fuera del primer corte.
